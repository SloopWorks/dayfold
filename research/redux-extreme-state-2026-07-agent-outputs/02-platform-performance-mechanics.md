# Agent memo 2/4 — platform performance mechanics at 10³–10⁴ action types

Raw fleet output for `research/2026-07-10-redux-extreme-state-architecture.md`.
Produced 2026-07-10 by a web-research agent; unedited. Claims labeled
`[fact:URL]` / `[estimate]` / `[assumption]`.

Scope: concrete costs/limits per platform for a Redux-style architecture
(single immutable state object, every mutation via dispatched action
classes/enum cases), at the scale of 10³–10⁴ action types.

---

## 1. Android / JVM / ART

**Dex reference limits.** A single DEX file can reference at most 65,536 methods (the `invoke-*` instructions take a 16-bit method index); fields and classes have the same 64K per-dex reference ceilings. Multidex removes the hard wall — with `minSdkVersion >= 21`, ART natively compiles multiple `classesN.dex` into one OAT file, and multidex is effectively free at runtime [fact:https://developer.android.com/build/multidex]. Practical implication: thousands of sealed action classes each add 1 class ref + constructor + `equals`/`hashCode`/`toString`/`copy`/`componentN` for data classes — [estimate] a data class contributes roughly 5–10 method refs, so 10,000 action data classes ≈ 50–100K method refs, i.e. enough to push a single-dex app over the limit by itself, but harmless post-multidex apart from size.

**R8/ProGuard.** R8's tree shaking, class merging (vertical + horizontal), and identifier renaming exist precisely to collapse this kind of long tail of small classes: "R8 combines sets of classes and interfaces into a single class, which reduces the number of classes in the app, lowering memory pressure"; typical whole-app size reductions are 10–65% [fact:https://developer.android.com/topic/performance/app-optimization/enable-app-optimization, fact:https://medium.com/androiddevelopers/shrinking-your-app-with-r8-909efac25de4]. Caveat [assumption]: action classes that are constructed reflectively/serialized (e.g., for devtools or persistence) need keep rules, which defeats merging/renaming for exactly the classes you have the most of.

**Class-loading / startup.** Classes are verified/loaded lazily, but everything touched on the startup path (a giant `Action` hierarchy referenced by a root reducer usually is) costs JIT/interpretation on first run. Baseline Profiles exist because AOT-compiling the startup path improves speed of first-launch code execution by ~30% (documented) and 40–50% in some production reports [fact:https://developer.android.com/topic/performance/baselineprofiles/overview, fact:https://medium.com/@vibhor.kul007/improve-your-android-app-startup-time-by-40-50-using-baseline-profiles-1716554761aa]. [estimate] Per-class load cost is small (microseconds-scale each on modern ART), so 10K classes is a startup tax on the order of tens of ms, not seconds — the bigger effect is dex/oat size and page-cache footprint.

**GC pressure from immutable copies.** ART since Android 8 uses Concurrent Copying with per-thread bump-pointer TLAB allocation (allocation ≈ pointer bump, no sync), generational since Android 10; the single GC pause is small and constant w.r.t. heap size [fact:https://source.android.com/docs/core/runtime/gc-debug, fact:https://sonique6784.medium.com/arts-garbage-collector-strategies-for-best-performance-1f5c79208851]. Consequence: short-lived action objects and shallow state copies are the *best case* for this GC (die young in the nursery). [estimate] The real risk is not action allocation but naive deep copies of large state on every dispatch — an O(state) copy per dispatch at 60 dispatches/s multiplies allocation rate by orders of magnitude; with structural sharing (copy only the changed path, `copy()` on data classes shares sub-objects by reference) the per-dispatch garbage is small. [assumption] For a dashboard-scale app (hundreds of KB of state, a few dispatches/s), GC is a non-issue; it becomes measurable when dispatch frequency approaches frame rate.

**Compose recomposition mechanics.** Compose's snapshot system registers observers on every state read in composition and invalidates only scopes that read a changed object [fact:https://developer.android.com/develop/ui/compose/state, fact:https://blog.zachklipp.com/introduction-to-the-compose-snapshot-system/]. Skipping is equality-based: with strong skipping (default since Compose compiler 2.0), stable params are compared with `equals()`, **unstable params with instance equality `===`** [fact:https://developer.android.com/develop/ui/compose/performance/stability/strongskipping]. This is the crux for a single-store design: if you pass the whole `AppState` (or an unstable slice) down the tree, its reference changes on *every* dispatch, so `===` always fails and nothing skips. Mitigations are exactly the documented ones: select narrow stable slices, `derivedStateOf` for frequently-changing inputs (recomposes only when the *derived* value changes), and deferring state reads to the lowest scope [fact:https://developer.android.com/develop/ui/compose/performance/stability/fix, fact:https://proandroiddev.com/conscious-compose-optimization-e16144b80eef]. [estimate] So "one immutable store" is fine iff subscription is selector-based with `distinctUntilChanged`-style value equality; it is pathological if the root state object is the recomposition input.

**Main thread.** Frame budget is 16ms at 60Hz (11ms @90, 8ms @120); frames >700ms are "frozen" [fact:https://developer.android.com/topic/performance/vitals/render]. Reducer work + subscriber notification per dispatch must fit inside the leftover budget after measure/layout/draw. Redux-Kotlin itself documents that multiplatform stores need a threading policy: same-thread-enforced store (throws `IllegalStateException` off-thread), deprecated fully-synchronized store, or the recommended concurrent store with lock-free reads and serialized writes [fact:https://reduxkotlin.org/introduction/threading, fact:https://reduxkotlin.org/api/createsamethreadenforcedstore]; background dispatch is possible if reducers get heavy [fact:https://patrickjackson.dev/redux-on-multithreaded-platforms/].

**Kotlin `when` over huge sealed hierarchies.** `when` over sealed *classes* compiles to a linear chain of `instanceof` checks, not `tableswitch`; only enums/ints get jump tables [fact:https://discuss.kotlinlang.org/t/does-when-is-with-sealed-classes-encourage-a-performance-antipattern/18044]. The measured gap is small (~2.1M vs ~2.3M ops/sec sealed-when vs enum-when in one benchmark) and "only matters when comparing thousands of constants in a loop" [fact:same thread/summary above]. [estimate] But at 10⁴ cases in *one* `when`, average match depth ~N/2 ⇒ thousands of type checks per dispatch — the standard fix is hierarchical dispatch (match on a parent sealed interface / feature sub-hierarchy first), which restores O(depth) behavior. [assumption] Kotlin compile time for one file containing a 10⁴-branch `when` will be painful (exhaustiveness checking + a single huge method possibly exceeding JVM's 64KB-bytecode-per-method limit); splitting per-feature reducers is both the idiomatic and the mechanical fix.

## 2. iOS / Swift

**Compile time.** Swift's constraint-solver type checker is the known bottleneck and can go super-linear/exponential; documented pathologies include a 12-line string-concatenation expression taking 42s to compile (vs 0.19s after a fix) and a one-line arithmetic expression taking 6.2s [fact:https://danielchasehooper.com/posts/why-swift-is-slow/]. Enums specifically: Swift 5.6 had an exponential compile-time regression on enums mixing cases with/without associated values [fact:https://forums.swift.org/t/large-increase-in-compilation-time-in-swift-5-6-on-enums-with-associated-values/56115]. TCA — the canonical "one enum of actions per feature, switch exhaustively" architecture — mitigates via aggressive modularization: isowords is split into 86 modules explicitly to improve compile times and SwiftUI-preview stability [fact:https://github.com/pointfreeco/isowords]. [estimate] A single flat `AppAction` enum with thousands of associated-value cases is a worst case for both the type checker and exhaustiveness checking; nested per-feature enums (TCA's pattern) keep each switch small. Use `-warn-long-expression-type-checking=100` to find hot expressions [fact:https://danielchasehooper.com/posts/why-swift-is-slow/].

**Binary size / metadata.** Every type/protocol pair emits a conformance record (`__swift5_proto`), and protocol-conformance lookup cost and metadata size grow with type count [fact:https://www.emergetools.com/blog/posts/SwiftProtocolConformance]. Enum value-witness tables carry tag/payload operations per enum [fact:https://belkadan.com/blog/2020/10/Swift-Runtime-Enums/]. [estimate] Thousands of action cases conforming to `Equatable`/`Sendable`/`Codable` multiply conformance records and synthesized `==` bodies — this is a real binary-size and link-time line item, though usually single-digit MB, not a blocker.

**Value-type state / CoW.** Swift structs copy eagerly; only stdlib collections are CoW. Large enum/struct payloads also inflate stack usage in accessors [fact:https://forums.swift.org/t/struct-and-enum-accessors-take-a-large-amount-of-stack-space/63251]. TCA's own Performance doc: sending actions "is not as lightweight of an operation as calling a method on a class" (each send traverses the whole reducer/effect machinery), share logic via methods not actions, and "high-frequency actions, such as sending dozens of actions per second, should be avoided" [fact:https://pointfreeco.github.io/swift-composable-architecture/1.1.0/documentation/composablearchitecture/performance/]. That is the strongest first-party statement that per-keystroke/per-frame dispatch through a composed store is an anti-pattern even in the flagship Swift Redux-alike.

**SwiftUI observation.** Old `ObservableObject`: any `@Published` change fires `objectWillChange` for the whole object → every subscribed view re-evaluates ("objectWillChange storm" for a single store object holding all app state). The iOS 17 Observation framework tracks *property-level* reads and invalidates only views that read the changed property [fact:https://fatbobman.com/en/posts/mastering-observation/]. [estimate] A single-store app on `ObservableObject` re-renders the world per dispatch; on `@Observable` it's fine only if state is decomposed into separately-tracked properties rather than one monolithic `var state: AppState` (a whole-struct replacement invalidates every reader of any field).

**Kotlin/Native (KMP store on iOS).** The new memory manager is a shared-heap concurrent mark/sweep GC (non-generational, non-moving), GC on its own thread, concurrent marking to shrink pauses; pauses are observable via Instruments signposts, and allocation spikes can force a stop-the-world phase [fact:https://kotlinlang.org/docs/native-memory-manager.html]. [estimate] Non-generational + non-moving means a high rate of short-lived action/state copies is *more* expensive on K/N-iOS than on ART — allocation-heavy dispatch loops hit K/N's weakest GC configuration. The old "frozen object / main-thread confinement" regime is gone (objects are shareable across threads) [fact:https://kotlinlang.org/docs/native-memory-manager.html], but UI updates still need marshaling to the main queue (`Dispatchers.Main`), and Kotlin↔Swift boundary crossings box/bridge values; Redux-Kotlin's concurrent store is the documented pattern [fact:https://reduxkotlin.org/introduction/threading].

## 3. Web / JS / Wasm

**V8 shape dispatch.** Property-access sites cache hidden classes: monomorphic (1 shape) → fast direct load; polymorphic (2–4) → linear check chain; **megamorphic (>4) → the optimizer gives up on that site** [fact:https://dl.acm.org/doi/pdf/10.1145/3428242, fact:https://braineanear.medium.com/the-v8-engine-series-iii-inline-caching-unlocking-javascript-performance-51cf09a64cc3]. Consequence for "many action shapes": any code that reads `action.payload`/`action.x` generically across thousands of differently-shaped action objects goes megamorphic. Mitigation [estimate]: the FSA convention (`{type, payload, meta}` — every action the same shape) keeps dispatch-path property reads monomorphic; class-per-action (Kotlin/JS or TS classes with distinct fields) does not.

**Large switches.** Jump tables only apply to dense integer cases; string-keyed switches degrade to sequential comparison, ~N/2 comparisons on average [fact:https://medium.com/@artemkhrenov/understanding-switch-vs-if-in-javascript-a-deep-dive-into-v8-optimizations-2d7c1ca61f9d]. [estimate] Redux's actual pattern — object-map lookup (`handlers[action.type]`, what `createSlice` does) — is O(1) and sidesteps this entirely; a literal 10⁴-case `switch` in one reducer is the only losing formulation.

**Immutability cost.** Immer (proxies) is ~2–3× slower than a hand-written spread reducer in its own worst-case benchmark (50K todos, 5K updates), roughly as fast as ImmutableJS, and its ES5 fallback ~2× slower again; it sometimes *wins* by detecting no-op updates and returning the original reference [fact:https://immerjs.github.io/immer/performance/]. There's an open RTK issue reporting 100–400× slowdowns vs vanilla Redux in specific pathological benchmarks [fact:https://github.com/reduxjs/redux-toolkit/issues/4793]. Immutable.js gets memory efficiency from HAMT/vector tries with 32-way branching (structural sharing: appending to a 1000-element vector doesn't copy it) [fact:https://immutable-js.com/, fact:https://medium.com/@dtinth/immutable-js-persistent-data-structures-and-structural-sharing-6d163fbd73d2] — but conversion to plain JS objects at component/network boundaries adds substantial overhead [fact:https://immerjs.github.io/immer/performance/].

**Redux DevTools.** DevTools serializes every action and a state snapshot into its history; large states make serialization slow and memory-heavy — documented as warnings and crashes [fact:https://github.com/zalmoxisus/redux-devtools-extension/issues/658, fact:https://github.com/pmndrs/zustand/issues/926]. Controls: `maxAge` (max retained actions; history is effectively a ring buffer — oldest actions dropped), `actionSanitizer`/`stateSanitizer` to strip bulky slices [fact:https://github.com/zalmoxisus/redux-devtools-extension/blob/master/docs/API/Arguments.md].

**Kotlin/Wasm size.** Baseline is class-count-sensitive: the compiler deliberately omits fully-qualified class names from the binary because storing them "increases the application size" (enabling `qualifiedName` is an opt-in size cost), and 2.2.20 added a `smallBinary` (-Oz) option; `wasm-opt` removes another 10–20% [fact:https://kotlinlang.org/docs/whatsnew2220.html, fact:https://www.infoq.com/articles/six-ways-optimize-webassembly/]. [estimate] Each Kotlin class carries vtable/itable/RTTI structures in Wasm-GC, so 10K action classes contribute linearly to module size; unreferenced cases are DCE-eligible, but a central exhaustive `when` references all of them, defeating DCE by construction.

## 4. Concurrency: dispatch as a serialization point

- The store serializes all writes by definition (JS: event loop; Kotlin: same-thread enforcement / reentrant-lock concurrent store [fact:https://reduxkotlin.org/introduction/threading]). Every dispatch runs root reducer + all subscriber notifications synchronously.
- **First-party guidance says don't put high-frequency ephemeral values through it.** Redux FAQ: "Dispatching an action on every keystroke of a text input probably isn't worthwhile… buffer keystrokes"; some data ("current value of a form input", "is this dropdown open?") doesn't belong in the store [fact:https://redux.js.org/faq/organizing-state, fact:https://redux.js.org/faq/performance]. React docs: values only needed by event handlers whose change shouldn't re-render (scroll position being the canonical example) belong in refs, not state [fact:https://react.dev/learn/referencing-values-with-refs]. TCA: avoid dozens-of-actions-per-second [fact:https://pointfreeco.github.io/swift-composable-architecture/1.1.0/documentation/composablearchitecture/performance/]. Historical react-redux issue: hundreds of rapid dispatches caused huge slowdowns pre-batching [fact:https://github.com/reduxjs/react-redux/issues/263].
- **Batching:** React 18 batches all updates (including timeouts/promises/native handlers) under `createRoot`, `flushSync` opts out [fact:https://github.com/reactwg/react-18/discussions/21]; `redux-batched-actions` packs N actions into one reducer pass [fact:https://redux.js.org/faq/performance]; RTK's `autoBatchEnhancer` delays subscriber notification to the next `requestAnimationFrame` for tagged actions [fact:https://redux-toolkit.js.org/api/autoBatchEnhancer]; saga/RxJS `throttle`/`debounce` for input streams [fact:https://redux-saga.js.org/docs/recipes/].
- **Lock-free/thread-confined stores:** Redux-Kotlin offers same-thread-enforced (confinement) and concurrent (lock-free reads, serialized writes) variants; full synchronization is deprecated because it blocked reads during dispatch [fact:https://reduxkotlin.org/introduction/threading]. [estimate] Pattern that works in practice: confine dispatch to the main/UI thread, run effects off-thread, and keep per-frame signals (scroll, animation, sensors) in local platform state (Compose `mutableStateOf` in the composable, SwiftUI `@State`, refs), dispatching only their *settled* results.

## 5. Memory: history & structural sharing

- Time-travel devtools retain (action, state-snapshot) pairs; unbounded history grows linearly with dispatch count × state size. `maxAge` bounds it ring-buffer-style; sanitizers cut per-entry size [fact:https://github.com/zalmoxisus/redux-devtools-extension/blob/master/docs/API/Arguments.md, fact:https://github.com/zalmoxisus/redux-devtools-extension/issues/658].
- With structural sharing, retained snapshots share unchanged subtrees, so N snapshots of a state where each dispatch touched one small path cost O(N × delta), not O(N × state) — the trie math (32-way branching, ~6 levels for 10⁹ elements) is what makes persistent-structure history cheap [fact:https://medium.com/@dtinth/immutable-js-persistent-data-structures-and-structural-sharing-6d163fbd73d2]. [estimate] Plain JS/Kotlin `copy()`-based reducers get the same sharing *if* they only spread along the changed path; devtools *serialization* (JSON) destroys sharing — the serialized history is O(N × state) regardless, which is why serializing devtools chokes on large stores while in-memory history doesn't.
- [assumption] On mobile (ART/K-N), keeping full time-travel history in production is not viable memory-wise; a small ring buffer of recent actions (no state snapshots, recompute by replay) is the standard compromise.

## 6. Tooling

- **IDE:** Kotlin completion/analysis is a documented pain point at scale (YouTrack KTIJ-8163: "incredibly slow code completion… while Java runs super smoothly"); JetBrains' own guidance: heavy type inference overloads analysis, explicit types help, reduce indexing scope [fact:https://youtrack.jetbrains.com/issue/KTIJ-8163, fact:https://blog.jetbrains.com/kotlin/2021/06/kotlin-ide-performance/]. [estimate] A 10K-subtype sealed hierarchy makes every `when`-exhaustiveness inspection and completion popup enumerate the hierarchy — cost lands on the interactive path, not just the build. Xcode/SourceKit shares the type-checker cost profile documented above (42s single-expression pathologies) [fact:https://danielchasehooper.com/posts/why-swift-is-slow/]; SourceKit-LSP exhaustive-switch completion over big enums is itself a discussed feature/scaling topic [fact:https://forums.swift.org/t/exhaustive-switch-completion-from-sourcekit-lsp/71341].
- **Incremental compilation blast radius.** Kotlin IC tracks class-member-level dependencies and recompiles only classes depending on changed ABI members; classpath-snapshot IC (1.7+) improved cross-module avoidance [fact:https://blog.jetbrains.com/kotlin/2022/07/a-new-approach-to-incremental-compilation-in-kotlin/, fact:https://kotlinlang.org/docs/gradle-compilation-and-caches.html]. But: adding a case to a *central* sealed `Action` type is an ABI change to a type every feature module depends on and exhaustively switches over ⇒ every dependent module recompiles. [estimate] This makes a monolithic action type the single worst build-cache key in the codebase; the isowords answer (86 modules, per-feature action enums composed at the root) exists precisely to contain this [fact:https://github.com/pointfreeco/isowords]. [assumption] Same logic applies to Swift (whole-module ABI) and to Gradle configuration-cache/build-cache hit rates.
- **Codegen:** [estimate] If action types are code-generated (e.g., from an API schema), the generated module compiles once and caches well as long as it's a *leaf dependency* that changes rarely; regenerating it invalidates everything downstream, same blast radius as above.

## Bottom line (synthesis)

The architecture is viable on all four surfaces at *dashboard-scale dispatch rates* (a few dispatches/s, state ≤ a few hundred KB), provided four disciplines, each backed by first-party guidance: (1) **selector-based, value-equality subscriptions** — never feed the root state reference to Compose/SwiftUI/React directly [fact:https://developer.android.com/develop/ui/compose/performance/stability/strongskipping]; (2) **hierarchical action types + hierarchical dispatch** — per-feature sealed/enum sub-hierarchies, never one flat 10⁴-case type (compile time, exhaustiveness, IC blast radius, `instanceof` chains) [fact:https://github.com/pointfreeco/isowords]; (3) **keep per-frame/per-keystroke signals out of the store** (Redux FAQ, React refs, TCA performance doc all converge) [fact:https://redux.js.org/faq/organizing-state, fact:https://react.dev/learn/referencing-values-with-refs, fact:https://pointfreeco.github.io/swift-composable-architecture/1.1.0/documentation/composablearchitecture/performance/]; (4) **bound devtools history and strip state snapshots** in serialized form [fact:https://github.com/zalmoxisus/redux-devtools-extension/blob/master/docs/API/Arguments.md]. The hard platform limits (64K dex refs, megamorphic ICs, Swift type-checker blowups, K/N non-generational GC) are all avoidable by structure, none by tuning.
