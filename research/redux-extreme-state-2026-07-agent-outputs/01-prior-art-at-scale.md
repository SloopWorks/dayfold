## 1. Elm at scale (NoRedInk)

**Scale numbers.** NoRedInk's main monolith holds **1,506 Elm files / 211,835 LOC** of production Elm in `monolith/ui/src`, plus **569 test files / 200,586 LOC** of Elm tests, **129 files / 26,399 LOC** of generated Elm, and another 80 files / 13,941 LOC in a secondary Haskell-backed service — and they run **"over a hundred Elm apps"** each serving a different Rails controller [fact:https://juliu.is/elm-at-noredink/]. Earlier public figures cite "80,000+ lines of production Elm" (2017 era) [fact:https://qconlondon.com/london-2017/london-2017/presentation/full-scale-elm-production.html].

**Crucial structural point: NoRedInk does NOT run one global store.** The 100+ Elm apps are separate TEA programs per Rails page, each with its own Model/Msg/update; a shared `Nri.Program` wrapper handles cross-cutting boilerplate, and view functions take an `Env` record for user/environment context [fact:https://juliu.is/elm-at-noredink/]. Where nesting exists it's shallow (e.g. `Teach.Classes.Main`'s Msg wraps `ModalMsg Modal.Msg`, `ClassesMsg Classes.Msg`) [fact:https://juliu.is/elm-at-noredink/].

**What broke / anti-patterns identified.** The community consensus, driven by NoRedInk's Richard Feldman ("Scaling Elm Apps", Elm Europe 2017), is that **nested-TEA "components" are the failure mode, not the scaling tool**: "If you choose nesting TEA applications as your default approach to scale Elm code, you will find yourself constantly wrapping and unwrapping messages for little or no benefit" (Ju Liu, NoRedInk) [fact:https://juliu.is/elm-at-noredink/]; Feldman's talk teaches avoiding mini update functions whose (model, cmd) results must be mapped back into the parent, using `Html.map` and "teach me how to message" (message-constructor-passing) only when genuinely reusable [fact:https://youtu.be/DoA4Txr4GUs] [fact:https://medium.com/elmupdate/summarising-elm-scaling-strategy-a942e7ee31c5]. The official Elm guide is blunt: "actively trying to make components is a recipe for disaster in Elm"; a sidebar should be a `viewSidebar` function, not its own Model/update; grow long files ("push your limit on number of lines") and split around data types, not screens/layers [fact:https://guide.elm-lang.org/webapps/structure]. NoRedInk correspondingly embraces large files (one 2,251-line file; several >1,500 lines) organized by comment sections [fact:https://juliu.is/elm-at-noredink/]. Feldman's original tutorial repo frames the whole topic as "how to create modular Elm code that scales nicely" precisely because naive Msg nesting caused wrapper explosion [fact:https://github.com/rtfeldman/elm-architecture-tutorial]. Community discussion of large-codebase practice: Elm Radio ep. 50 (Large Elm Codebases with Ju Liu) [fact:https://elm-radio.com/episode/large-codebases/] and ep. 19 (Scaling Elm Apps) [fact:https://elm-radio.com/episode/scaling-elm-apps/].

**Takeaway [assumption]:** the largest documented Elm shop scaled by *sharding into ~100 independent TEA programs* and by flattening (fewer Msg types, more plain functions), not by growing one Msg union to thousands of variants.

## 2. Redux at very large scale

**Twitter Lite** (one of the largest React PWAs; Node/React/Redux/Normalizr/combineReducers): the single most-cited scaling pathology was **dispatch cost of high-frequency actions into the global store** — moving draft-tweet state out of Redux (a dispatch per keypress) into local component state "reduced overhead by over 50%" in the composer flow [fact:https://medium.com/@paularmstrong/twitter-lite-and-high-performance-react-progressive-web-apps-at-scale-d28a00e780a3]. They kept Redux but normalized state (Normalizr) and split it into store areas via combineReducers; rendering-side fixes (virtualized lists, rAF/requestIdleCallback deferral) were separate [fact:same URL] [fact:https://blog.x.com/engineering/en_us/topics/open-source/2017/how-we-built-twitter-lite].

**Documented action-type counts.** The official Redux FAQ (Performance) cites a real production report of **~500 action types, ~400 reducer cases, ~150 components, 5 middlewares, ~2,300 tests** performing "very well" [fact:https://redux.js.org/faq/performance]. Same page: reducer combination is cheap ("JS engines run a very large number of function calls per second… most reducers just hit a switch default"), and the recommended defense against re-render storms is **many small connected components + memoized selectors** (connect list items individually by ID) rather than a few big `mapStateToProps` [fact:https://redux.js.org/faq/performance]. Note the FAQ's documented comfort zone is *hundreds*, not thousands, of action types [estimate].

**DevTools/serialization pathology.** With large states/action payloads, Redux DevTools emits "Application state or actions payloads are too large making Redux DevTools serialization slow and consuming a lot of memory"; documented mitigations are actionSanitizer/stateSanitizer or excluding store slices [fact:https://github.com/zalmoxisus/redux-devtools-extension/issues/658] [fact:https://github.com/pmndrs/zustand/issues/926] [fact:https://dev.to/migsarnavarro/use-sanitizers-to-avoid-redux-devtools-crash-67p]. i.e., the time-travel/inspectability benefit that motivates single-store degrades first at scale [assumption].

**Jira (Atlassian).** Redux was Jira's frontend state standard, but Jira in practice ran **many parallel Redux providers ("every app in Jira had a Redux provider at the top")**, and cross-provider communication became the blocker; consolidating into one true global store "seemed like a herculean effort, definitely not deliverable in a few months" — so Atlassian built **react-sweet-state** (selector-based re-renders + Redux-devtools compatibility + multiple small stores) instead [fact:https://medium.com/@albertogasparin/react-sweet-state-redux-and-context-the-yummy-parts-f55f49503635] [fact:https://github.com/atlassian/react-sweet-state]. Their router is built on it [fact:https://www.atlassian.com/blog/atlassian-engineering/react-resource-router-deep-dive]; a talk documents the broader migration ("Jira Frontend architecture (r)evolution") [fact:https://hasgeek.com/jsfoo/2019/sub/jira-frontend-architecture-r-evolution-a-story-of-6N4Mc38EMk5nMqJnk3wv6H]. So the *documented* large-Redux reality at Atlassian was already federated stores, and the fix was more federation with better sharing — not one store [assumption based on above facts].

**Why teams left / alternatives emerged.** Facebook's Recoil was built after hitting "a performance and efficiency wall" with existing global-state approaches on a complex internal tool; core criticism: the Redux store lives *outside* React's scheduler, and Recoil's atom graph gives per-atom subscription granularity for apps rendering very many interdependent components [fact:https://medium.com/swlh/recoil-another-react-state-management-library-97fc979a8d2b] [fact:https://www.imaginarycloud.com/blog/recoil-vs-redux]. Jotai's author states his motivation was "to avoid Redux selectors (mapStateToProps)" because referential-equality-based selector correctness is hard for teams, plus removing Recoil's string keys [fact:https://blog.axlight.com/posts/how-jotai-was-born/]. Zustand (Poimandres) positions itself as removing Redux's reducer/action ceremony and Context's re-render problem via a subscription model [fact:https://github.com/pmndrs/zustand] [fact:https://zustand.docs.pmnd.rs/].

**Redux's own response.** Redux Toolkit's `createSlice` exists explicitly because hand-written action types/creators were the #1 complaint: it "automatically generates action creators and action types that correspond to the reducers and state," and is now "the standard approach for writing Redux logic" [fact:https://redux-toolkit.js.org/api/createSlice] [fact:https://redux-toolkit.js.org/usage/usage-guide]. RTK Query and listener middleware further remove hand-rolled fetch/side-effect action triples [fact:https://redux-toolkit.js.org/usage/usage-guide]. Net: the maintainers' answer to action explosion was *generation + co-location per slice*, keeping the single store but making action types an implementation detail [assumption grounded in the above].

## 3. re-frame (ClojureScript)

re-frame is the closest published analogue to "one app-db, explicit events" at scale, and its docs are unusually candid about the scaling mechanics:

- **Signal graph (layer 2 / layer 3).** Subscriptions form a DAG rooted at `app-db`: Layer-2 "extractors" pull data directly from app-db; Layer-3 "materialized views" compute derived data only from other subscriptions. Critically: **"all (currently instantiated) layer-2 subscriptions run every time app-db changes in any way… so we want them to be computationally trivial"** — the layering exists precisely to stop derivation cost from scaling with every event [fact:https://day8.github.io/re-frame/subscriptions/] [fact:https://github.com/day8/re-frame/blob/master/docs/subscriptions.md]. The re-frame-10x debugger even ships a feature to hide unchanged layer-2 subs because there are so many firing per event [fact:https://cljdoc.org/d/day8.re-frame/re-frame-10x/0.3.5/doc/ignoring-unchanged-layer-2-subscriptions].
- **Event queue as FSM.** The router's event queue is explicitly "modeled as a Finite State Machine" (states `:idle`, `:scheduled`, running, `:paused`), designed to drain event cascades in one burst without hogging the CPU or blocking redraw, with `:flush-dom` pauses for animation frames [fact:https://github.com/day8/re-frame/blob/master/src/re_frame/router.cljc].
- **Interceptors** replace middleware; the docs warn that in a big app "if every developer just adds a bunch of global interceptors, after a while your application would start to get pretty slow" [fact:https://github.com/Day8/re-frame/blob/master/docs/Interceptors.md].
- **Scale reports.** The official README claims use "by both small startups and companies with over 500 developers" and production apps of "40K lines of code and beyond" [fact:https://github.com/day8/re-frame]; Solita reports "the re-frame architecture scales very well if you are consistent in naming and separating responsibilities" [fact:https://dev.solita.fi/2024/02/02/the-things-i-like-about-cljs-and-re-frame.html]; a JUXT case study covers a data-heavy app handling "hundreds of UI changes per second" and the CLJS perf work needed [fact:https://www.juxt.pro/blog/clojurescript-app-performance/]. Guidance on structuring big app-dbs: [fact:https://github.com/day8/re-frame/blob/master/docs/App-Structure.md]. Early community scaling thread: [fact:https://groups.google.com/g/clojurescript/c/Ta8HnekpiZk].

**Takeaway [assumption]:** re-frame is the one ecosystem that kept the literal single store at scale, and it survived by (a) making event dispatch an async FSM queue, (b) enforcing a layered, memoized subscription DAG so O(events)×O(subscriptions) stays cheap, and (c) namespaced keyword events (no central enum, so no type-level "action explosion" — the cost moves to runtime discoverability).

## 4. Swift Composable Architecture (TCA)

TCA is the best-documented case of a *typed* single-store architecture hitting hard limits, because Swift enums make action explosion a compile-time object:

- **Official performance doc** (Point-Free): "Sending actions is not as lightweight of an operation as calling a method on a class" — each action "travels through multiple layers of the application"; sharing logic via actions is called out as an anti-pattern (extract `inout State` methods instead); high-frequency actions are explicitly discouraged (for 100,000 progress steps, "report the progress at most 100 times"); reducers run on the main thread so CPU-heavy work must move to effects; post-1.5 scoped stores compute child state on the fly, so heavy computed properties in scopes multiply cost [fact:https://raw.githubusercontent.com/pointfreeco/swift-composable-architecture/main/Sources/ComposableArchitecture/Documentation.docc/Articles/Performance.md].
- **Compile times.** Upgrading to TCA 1.7 (macros) produced reports of **~8x longer builds**, all cores >95% on an M3 Max, because Swift macros force compiling swift-syntax (15–20 s debug, 2+ min release); maintainers: "a well known problem with macros… Apple has not provided guidance" [fact:https://github.com/pointfreeco/swift-composable-architecture/discussions/2776]. Separately, large SwiftUI views over TCA state hit "the compiler is unable to type-check this expression in reasonable time" [fact:https://github.com/pointfreeco/swift-composable-architecture/discussions/2594].
- **Runtime pathologies (single root reducer tree).** Documented discussions: stack-overflow potential with large app states / deep reducer nesting [fact:https://github.com/pointfreeco/swift-composable-architecture/discussions/488]; send-recursion performance [fact:https://github.com/pointfreeco/swift-composable-architecture/discussions/1290]; slow child-action propagation [fact:https://github.com/pointfreeco/swift-composable-architecture/discussions/382].
- **Three-year production retrospective (Rod Schmidt, 8 teams on one codebase):** "Our application reducer was so large, Xcode had problems scrolling through it, and it wouldn't give you valid compiler errors" — they split the app reducer across **~20 file extensions**; action processing "can be quite slow — each action has to go up through the entire reducer tree"; they had to **increase allocated stack space** to avoid overflows; stack traces became unusable (framework frames swamp app frames); no encapsulation ("any parent reducer can get access to any child's state or actions") caused inter-team coupling [fact:https://rodschmidt.com/posts/composable-architecture-experience/]. This is the clearest documented instance of the "what happens to build times/IDE/team ergonomics" failure profile in a typed single store.
- **Reference app.** isowords: full production game, client+server, **91 SPM modules, ~50k LOC**, entire app on TCA — notably scaled by aggressive *modularization* (average 70 LOC per app target; features as separate modules) rather than one flat action tree [fact:https://github.com/pointfreeco/isowords] [fact:https://www.pointfree.co/blog/posts/55-open-sourcing-isowords].

## 5. Android MVI at scale — all major frameworks chose per-screen stores

- **Airbnb MvRx/Mavericks.** Built when Airbnb was adding **286 new screens in one year**, designed for "an app with 70 engineers and nearly a million lines of code"; state lives in `MavericksState` per ViewModel, scoped to the Fragment by default (`fragmentViewModel`), with `activityViewModel` only for cross-screen sharing — official guidance: scope to Fragment if state is screen-local [fact:https://medium.com/airbnb-engineering/introducing-mvrx-android-on-autopilot-552bca86bd0a] [fact:https://github.com/airbnb/mavericks/wiki]. The intro post gives no explicit anti-global-store argument; the rationale is implicit in the lifecycle/screen coupling [assumption]. Airbnb's successor architecture Trio continues per-screen composition [fact:https://medium.com/airbnb-engineering/introducing-trio-part-i-7f5017a1a903].
- **Spotify Mobius.** Explicitly loop-per-feature: "In a Spotify context, there is usually one loop per feature such as 'the album page', 'login flow', etc.", each loop owning its own Model/Events/Update/Effects; loops may also be UI-less and session-scoped [fact:https://github.com/spotify/mobius/wiki/Concepts].
- **Badoo/Bumble MVICore.** `Feature` (their Store) is a per-feature unit — "you can operate with a single Reducer if needed… with the full power of additional components for complex cases"; the architecture composes many Features rather than one store [fact:https://badoo.github.io/MVICore/features/coreconcepts/] [fact:https://badootech.badoo.com/a-modern-kotlin-based-mvi-architecture-9924e08efab1].
- **Slack Circuit.** "UDF-first, keyed by Screens": one Presenter per Screen produces that screen's state; events flow UI→presenter; used in production in Slack's Android app; influenced by Cash App's Broadway [fact:https://slackhq.github.io/circuit/] [fact:https://slackhq.github.io/circuit/states-and-events/] [fact:https://github.com/slackhq/circuit]. Talk: "Modern Compose Architecture with Circuit" (Sweers/Elliott, KotlinConf) [fact:https://www.youtube.com/watch?v=ZIr_uuN8FEw].

**Pattern [assumption]:** every large-scale mobile team that adopted Redux/Elm-style UDF (Airbnb, Spotify, Badoo/Bumble, Slack, Cash App) independently landed on *per-screen/per-feature state machines composed by a navigator*, not one app-wide store — mobile lifecycles, process death, and team ownership boundaries make a global store's costs dominate.

## 6. Single global store with 1,000+ action types — documented cases

No engineering blog documents a healthy app with 1,000+ action types in one typed store [estimate — extensive search found none]. What exists:

- **Redux official ceiling data point:** ~500 action types / ~400 reducer cases reported as fine (untyped JS, string constants — no IDE/type-check cost) [fact:https://redux.js.org/faq/performance].
- **TCA (typed) at ~app-wide-reducer scale:** the Rod Schmidt case above — Xcode scrolling/compiler-error breakdown, reducer split into ~20 extensions, stack-size increases, unusable traces [fact:https://rodschmidt.com/posts/composable-architecture-experience/]; plus the macro-era 8x build-time regression [fact:https://github.com/pointfreeco/swift-composable-architecture/discussions/2776]. This is the closest documented "what happens to build times/IDE/team ergonomics."
- **Indirect evidence:** NoRedInk sharded into 100+ programs rather than grow one Msg type [fact:https://juliu.is/elm-at-noredink/]; Jira never achieved one store and built sweet-state instead [fact:https://medium.com/@albertogasparin/react-sweet-state-redux-and-context-the-yummy-parts-f55f49503635]; re-frame avoids the problem entirely via keyword events (no enum) [fact:https://day8.github.io/re-frame/subscriptions/].

## Cross-cutting synthesis signals

1. **Nobody scales the literal "one store, one action enum" shape past low hundreds of action types.** Survivors either shard programs/stores per screen or feature (NoRedInk, Jira, all Android MVI, isowords-via-modules) or de-typify events and invest in a runtime dispatch/subscription engine (re-frame) [assumption from all facts above].
2. **The recurring runtime failure is O(actions × tree depth × subscribers):** every action traversing the whole reducer tree (TCA docs, Schmidt), every layer-2 sub firing per event (re-frame docs), keypress-dispatch overhead (Twitter Lite's 50%). Mitigations: throttle/coalesce high-frequency actions, keep ephemeral state local, memoized/layered selectors.
3. **The recurring build-time/ergonomics failure is typed-enum-specific** (Swift exhaustive switches, macros, type-checker), largely absent in JS/CLJS — relevant when choosing Kotlin sealed-class action hierarchies [assumption].
4. **Tooling (time-travel devtools) degrades before the architecture does** — DevTools serialization warnings are the canary [fact:https://github.com/zalmoxisus/redux-devtools-extension/issues/658].
