# Agent memo 6/6 — LLM difficulty gradients & model tiering for redux-style code

(operator asked: are reducers/selectors/UI=f(state) predictable enough for
cheaper LLMs?). Produced 2026-07-10 by a web-research agent; unedited.
Claims labeled `[fact:URL]` / `[estimate]` / `[assumption]`.

**Question:** Which kinds of code do LLMs generate reliably vs. unreliably, and does evidence support delegating pattern-dense, pure, cheaply-testable code (Redux-style reducers/selectors/state types) to smaller/cheaper models?

**Bottom line up front:** The evidence strongly supports a *difficulty gradient* (isolated/local/testable code is near-solved even for small models; cross-file/stateful/effectful code is where models fail) and supports *routing + verifier loops* as cost levers. The specific claim "pure/functional code is easier for LLMs" is **only partially supported** — purity of *style* within a mainstream language helps (testability, locality), but purely *functional languages* (Haskell/OCaml) are actually harder for LLMs due to training-data scarcity. The hypothesis should be framed as "locally-scoped, convention-dense, machine-checkable Kotlin" rather than "functional code."

---

## 1. Task-difficulty gradients: isolated-function vs repo-level

- **HumanEval/MBPP are saturated.** Frontier models score in the low 90s on HumanEval, making it useless for differentiating them; by contrast repo-level SWE-bench Verified remained hard for years (typical gap: 82–97% HumanEval vs 54–81% SWE-bench for the same model cohort) [fact:https://www.layer3labs.io/guides/ai-coding-benchmarks], [fact:https://benchmarkingagents.com/swe-bench/]. (Note: by mid-2026 the frontier tops ~87% on SWE-bench Verified under agent harnesses [estimate], so the gap is closing at the top — but it closes via *expensive frontier models with agentic scaffolding*, which is exactly the tier the hypothesis reserves them for.)
- **ClassEval is the cleanest controlled evidence.** Moving from standalone functions to a class with interdependent methods, GPT-4 drops from **85.4% (HumanEval) to 37.0% pass@1 (ClassEval)** — a ~48-point cliff — and even *method-level* pass@1 inside classes drops (GPT-4: 85.4%→62.5%; GPT-3.5: 68.9%→52.5%). The authors attribute the drop to dependency on surrounding context/state [fact:https://arxiv.org/abs/2308.01861], [fact:https://mingwei-liu.github.io/assets/pdf/ICSE2024ClassEval-V2.pdf].
- **Cross-file context is the failure axis.** CrossCodeEval (NeurIPS 2023) shows completion is "extremely challenging when the relevant cross-file context is absent" and improves sharply when it's supplied; even best-retrieval + best-model is far from ceiling [fact:https://arxiv.org/abs/2310.11248]. RepoBench splits the problem into retrieval (find cross-file snippets) and completion, confirming the same dependency [fact:https://crosscodeeval.github.io/].
- **Complexity-metric correlation is murky — semantic, not structural, difficulty dominates.** A feedback-driven study finds complexity metrics predictive of pass@1 via logistic regression [fact:https://arxiv.org/abs/2505.23953], and one HumanEval+/APPS+ study tracks cyclomatic complexity rising 3.1→6.9 alongside falling pass rates [fact:https://arxiv.org/html/2407.06153]. But a security-patch failure analysis found **no significant correlation** between LOC/cyclomatic complexity and LLM success — difficulty is "knowing what to change," not navigating complex code [fact:https://arxiv.org/html/2603.10072], and a benchmark meta-analysis found classical metrics have negligible R² on HumanEval/MBPP failures [fact:https://arxiv.org/html/2511.04355v1]. A 2026 paper proposes *LLM-perceived* complexity (LM-CC), which does correlate consistently [fact:https://arxiv.org/abs/2602.07882]. **Implication:** the tiering axis should be *non-locality + ambiguity* (which Redux reducers minimize), not cyclomatic complexity per se [assumption].

## 2. Purity/determinism specifically — the nuanced part

- **Functional *languages* are harder, not easier.** FPBench (721 tasks, Haskell/OCaml/Scala) finds error rates "significantly higher in purely functional languages than in hybrid or imperative": GPT-4-class ~21% compile-error rate in Haskell vs 6.24% Java; even GPT-5 stays at 24.28% Haskell vs 4.62% Java. Cause: training-data scarcity [fact:https://arxiv.org/abs/2601.02060v1]. A Haskell completion case study reaches the same conclusion [fact:https://arxiv.org/html/2403.15185v1]. **So "Elm-style purity in Kotlin" keeps the benefits (determinism, testability) without the data-scarcity penalty — Kotlin is mainstream-corpus** [assumption].
- **Property-based testing pairs well with LLM codegen.** PBT-as-validator exposes correctness gaps pass@k misses (30–32% of "passing" solutions only partially satisfy properties) [fact:https://arxiv.org/html/2506.18315v1], [fact:https://dl.acm.org/doi/10.1145/3696630.3728702]; LLM-generated PBT + example tests together raised bug detection from 68.75%→81.25% [fact:https://arxiv.org/abs/2510.25297]. Pure reducers are the ideal PBT target (deterministic, no mocks) [assumption].
- **Verifier loops help weaker models most.** "How Many Tries Does It Take?" (7 models, Llama 8B→Gemini 2.5 Pro): test-driven self-repair with ≤5 attempts universally improves pass rates — **+4.9 to +17.1 pts HumanEval, +16.0 to +30.0 pts MBPP — with gains concentrated in the first two rounds and largest for weaker models** [fact:https://arxiv.org/abs/2604.10508]. Caveat: repair loops stall without runtime insight (assertion/logic errors repair at only ~45%) [fact:https://arxiv.org/html/2602.06875v1]. Reducers' cheap deterministic tests give exactly the runtime signal repair loops need [assumption].

## 3. Small/cheap model capability

- **On isolated, patterned code, small models already match giants.** Qwen2.5-Coder-**7B** scores **88.4% HumanEval**, beating original GPT-4's 87.1% [fact:https://qwenlm.github.io/blog/qwen2.5-coder-family/], [fact:https://arxiv.org/abs/2409.12186]. The gap re-opens on repo-level/agentic tasks (SWE-bench), where 7B-class open models score far below frontier [estimate — consistent across leaderboards, e.g. https://www.codesota.com/code-generation].
- **Haiku-class:** Claude Haiku 4.5 scores **73.3% SWE-bench Verified at $1/$5 per MTok** — "similar coding performance to Sonnet 4 at one-third the cost and 2x+ speed," and ~90% of Sonnet 4.5 on Augment's agentic eval [fact:https://www.anthropic.com/news/claude-haiku-4-5]. So the "cheap tier" in 2026 is not toy-grade; it's last-year's frontier.
- **FIM/autocomplete in production is the existence proof.** The original Copilot ran on Codex **12B** (28.8% HumanEval pass@1 was enough for a useful product) [fact:https://arxiv.org/abs/2107.03374]; GitHub later shipped successively small custom completion models tuned for sub-200ms latency [fact:https://github.blog/ai-and-ml/github-copilot/the-road-to-better-completions-building-a-faster-smarter-github-copilot-with-a-new-custom-model/]. Cursor's Tab is a small custom RL-trained model serving 400M+ requests/day, and Cursor's stated strategy is "a big model makes high-level decisions, a small model fills out the details" [fact:https://cursor.com/product/tab], [fact:https://forum.cursor.com/t/question-about-cursors-tab-completion-model/80725]. Pattern-dense line/block completion — the closest production analog to "write the next reducer branch" — has always run on the cheap tier.

## 4. Model routing / cascades

- **FrugalGPT** (Stanford, 2023/TMLR 2024): LLM cascades match GPT-4 with **up to 98% cost reduction**, or beat GPT-4 by 4% at equal cost; key finding is that cheap models answer many queries correctly and a learned scorer detects the hard ones [fact:https://arxiv.org/abs/2305.05176].
- **RouteLLM** (Berkeley/LMSYS, ICLR 2025): preference-trained routers cut cost **>85% on MT-Bench, 45% MMLU, 35% GSM8K at 95% of GPT-4 quality**; with augmentation, 95% quality using only 14% strong-model calls [fact:https://lmsys.org/blog/2024-07-01-routellm/], [fact:https://github.com/lm-sys/routellm].
- **aider's architect/editor split** is the code-specific production analog: strong reasoner plans, cheap model emits edits; o1-preview architect + DeepSeek/o1-mini editor set the then-SOTA 85% on aider's editing benchmark — evidence that *edit-emission* (mechanical, format-bound) is separable and delegable [fact:https://aider.chat/2024/09/26/architect.html].
- **Claude Code itself** defaults subagent/simple-task work to Haiku; third-party measurements of three-tier routing (Opus→architecture, Sonnet→implementation, Haiku→quick edits/review) report ~$0.98 vs $2.02 per session vs uniform Opus, with the largest savings from routing quick edits to Haiku ($0.66→$0.14) [fact:https://www.mindstudio.ai/blog/manage-token-costs-claude-code-dynamic-workflows], [estimate — vendor-blog numbers, not peer-reviewed].

## 5. Context economy

- **Lost in the Middle** (Liu et al., TACL 2024): U-shaped accuracy — models reliably use info at context start/end, degrade sharply mid-context, even explicit long-context models [fact:https://arxiv.org/abs/2307.03172].
- **RULER**: despite perfect vanilla needle-in-haystack, *all* 17 tested models degrade on harder long-context tasks as length grows; most models lose **15–30% accuracy between 4K and 128K**, and smaller/alternative architectures (Mamba-2.8B, RWKV) collapse past ~4–8K [fact:https://arxiv.org/abs/2404.06654]. Effective context < advertised context, and the shortfall is worse down-market [estimate — RULER shows claimed-vs-effective gaps correlate with model tier].
- **Implication for "10k-line flat Model.kt":** long flat files consume the cheap model's effective context on irrelevant material and push relevant declarations into the degraded middle. Slicing state into small per-feature reducer files is a *context-economy* optimization that disproportionately benefits the cheap tier [assumption, grounded in the above facts].

## 6. Constrained/schema-bound generation

- Grammar-constrained decoding guarantees syntactic validity but is not free: on small models (1–9B), constrained decoding adds **3.6–8.2x latency and can substantially degrade task accuracy** ("constraint tax") because rigid schemas suppress the model's intermediate reasoning [fact:https://arxiv.org/abs/2605.02363], [fact:https://arxiv.org/pdf/2605.26128]. Draft-then-constrain (DCCD) recovers much of it — a 1B model improved 15.24%→39.0% strict accuracy on JSON-GSM8K [fact:https://arxiv.org/html/2603.03305]. JSONSchemaBench benchmarks validity across engines/models [fact:https://arxiv.org/pdf/2501.10868].
- **Relevance:** sealed-class + exhaustive-`when` Kotlin is a *compiler*-constrained task rather than decoder-constrained — the verifier (kotlinc exhaustiveness check) runs post-hoc, avoiding the constraint tax while keeping the guarantee. That's the cheaper pattern the literature favors: generate freely, verify mechanically, repair [assumption + fact-pattern from §2 repair-loop results].

## 7. Redux/Elm/MVI + AI agents specifically

- **reduxkotlin.org has an explicit AI-agents section** (https://reduxkotlin.org/ai-agents/building-with-ai-agents) positioning redux-kotlin as "agent-ready": unidirectional architecture + deterministic patterns "reduce the token cost and iteration cycles agents need to produce working code," with an AGENTS.md "token-tight index" and "correct-by-default patterns" yielding "fewer write→verify cycles" [fact:https://reduxkotlin.org/ai-agents/building-with-ai-agents]. This is advocacy, not measurement — no benchmark numbers published [assumption: directionally right but unquantified]. (Directly relevant to Dayfold, which uses redux-kotlin.)
- Beyond that, peer-reviewed evidence on unidirectional UI architectures + LLM codegen specifically is **absent** — the closest support is indirect: ClassEval's state-dependency cliff (§1), locality/context findings (§5), and verifier-loop results (§2). "Agentic Redux" papers apply Redux ideas to multi-agent systems, not codegen [fact:https://redis.io/blog/ai-agent-architecture/ — tangential].

## Synthesis for the hypothesis

**Supported:** (a) isolated, deterministic, cheaply-testable code sits in a class small models handle at frontier-equivalent rates (Qwen-7B ≈ GPT-4 on HumanEval; Haiku 4.5 ≈ Sonnet 4 on coding); (b) failure concentrates where reducers/selectors deliberately have nothing — cross-file dependency, hidden state, effects (ClassEval −48 pts; CrossCodeEval); (c) cheap-model + test-driven repair loops converge fastest exactly when tests are deterministic and fast (+16–30 pts on MBPP-class tasks, gains largest for weak models); (d) routing/cascading is proven at 75–98% cost reduction with quality retention (FrugalGPT, RouteLLM), and every production coding product already tiers (Copilot, Cursor, aider, Claude Code).

**Corrected/qualified:** don't argue from *functional languages* — Haskell/OCaml are empirically LLM-hostile (4–5x the error rate of Java, even for GPT-5) because of corpus scarcity; the winning framing is *pure-style code in a mainstream language*. Also, structural complexity metrics don't predict failure well — **non-locality and specification ambiguity do** — so route on "does this task touch effects/other files/unclear requirements," not on cyclomatic complexity. And avoid decoder-level grammar constraints on small models (constraint tax); prefer generate → compile/test verify → bounded repair.

**Biggest evidence gap:** no published head-to-head of small-vs-large models *specifically on reducer/selector-style codegen*. Given Dayfold's redux-kotlin codebase and cheap test loop, that's a directly runnable in-house experiment (Haiku vs Sonnet on reducer tasks with the existing test suite as verifier) [assumption/recommendation].
