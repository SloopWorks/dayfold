## 1. Backend event-sourcing lessons that transfer

**The pattern and its dual reading.** Fowler defines event sourcing as storing "all changes to application state … as a sequence of events," giving audit trails, temporal queries, replay-based error correction, and the ability to reproduce production bugs by replaying real events in a test environment [fact:https://martinfowler.com/eaaDev/EventSourcing.html]. Crucially, Fowler notes the log can be either the official record with derived application state, or vice versa — the choice is architectural, not intrinsic [fact:https://martinfowler.com/eaaDev/EventSourcing.html]. Greg Young's canonical framing (Code on the Beach 2014, CQRS documents) adds that an event log — unlike a plain transaction log — **captures intent** [fact:https://www.kurrent.io/blog/transcript-of-greg-youngs-talk-at-code-on-the-beach-2014-cqrs-and-event-sourcing; fact:https://cqrs.files.wordpress.com/2010/11/cqrs_documents.pdf].

**CQRS is the read/write split, and it's a scoped tool.** Fowler: CQRS lets you "use a different model to update information than the model you use to read information," pairs naturally with event sourcing, but "for most systems CQRS adds risky complexity" and should apply only to specific bounded contexts [fact:https://martinfowler.com/bliki/CQRS.html]. Transfer to the client: the event log (actions) is the write model; the store/selectors/materialized views are derived read models that can be rebuilt or reshaped freely.

**Snapshotting.** Snapshots are a performance optimization, not an architectural requirement; Young says he wouldn't consider snapshotting until roughly a thousand events per stream [fact:https://codeopinion.com/greg-young-answers-your-event-sourcing-questions/]. Kurrent (EventStoreDB's company) guidance: implement snapshots only when measured replay latency is actually unsatisfactory; loading dozens of small events is not significant overhead [fact:https://www.kurrent.io/blog/snapshots-in-event-sourcing/; fact:https://domaincentric.net/blog/event-sourcing-snapshotting]. EventStoreDB itself has no built-in snapshotting — the idiom is appending state snapshots to a side stream every N events [fact:https://domaincentric.net/blog/event-sourcing-snapshotting]. Client translation: a persisted materialized store (e.g. serialized state or SQLite db) *is* the snapshot; you replay only the tail.

**Log compaction.** Kafka's compaction retains at least the last record per key and treats a null-payload record as a tombstone (deleted after `delete.retention.ms`) — i.e., a principled way to bound a log when only latest-value-per-key matters [fact:https://docs.confluent.io/kafka/design/log_compaction.html; fact:https://kafka.apache.org/30/generated/topic_config.html]. LiveStore has adopted exactly this design for client eventlogs: planned "eventlog compaction" where a new event for a given key drops previous events for that key, explicitly to avoid "unbounded growth of mutation eventlog" [fact:https://github.com/livestorejs/livestore/issues/136].

**Schema evolution / upcasting.** The dominant strategy is **upcasting**: middleware converting old event versions to the current version at deserialization/replay time; additive changes are non-breaking, and a new event version must remain convertible from the old one [fact:https://www.infoq.com/news/2017/07/versioning-event-sourcing/; fact:https://event-driven.io/en/how_to_do_event_versioning/]. Young wrote an entire book on this ("Versioning in an Event Sourced System", https://leanpub.com/esversioning) — his hard case is two software versions running concurrently against one log [fact:https://www.infoq.com/news/2017/07/versioning-event-sourcing/]. An empirical study of industrial event-sourced systems found schema evolution is the recurring pain point practitioners underestimate [fact:https://www.sciencedirect.com/science/article/pii/S0164121221000674]. Client implication: if actions are persisted (for sync or replay-in-CI), **action schemas become a versioned public contract** the moment one is written to disk — this is the single biggest hidden cost of the "extreme" position [assumption, grounded in the above].

## 2. The new wave: client sync engines are literally client-side event sourcing

**LiveStore (Johannes Schickling)** is the purest embodiment: "similar to Redux but persisted and synced across devices" [fact:https://dev.docs.livestore.dev/evaluation/event-sourcing/]. Events go into an immutable, ordered eventlog (the canonical history); **materializers** — event-handler functions mapping events to SQL state changes — project them into a reactive local SQLite database; both eventlog and materialized state are persisted, so startup does not replay the whole log; rematerialization from the eventlog happens on schema migration ("auto" migration strategy) [fact:https://dev.docs.livestore.dev/reference/concepts/; fact:https://dev.docs.livestore.dev/evaluation/how-livestore-works/; fact:https://docs.livestore.dev/reference/state/materializers/]. Sync is git-like push/pull with a global total order: local unpushed events are **rebased** on top of upstream events before pushing [fact:https://dev.docs.livestore.dev/evaluation/how-livestore-works/; fact:https://docs.livestore.dev/reference/syncing/]. LiveStore's own stated tradeoffs: more boilerplate defining events, and "be careful so eventlog doesn't grow too much" [fact:https://dev.docs.livestore.dev/evaluation/event-sourcing/]. It distinguishes **synced** vs **client-only** events — an explicit two-tier log [fact:https://dev.docs.livestore.dev/reference/concepts/]. Schickling's talk "Sync different: Event sourcing in local-first apps" (local-first conf) is the narrative version [fact:https://www.youtube.com/watch?v=nyPl84BopKc].

**Replicache/Zero (Rocicorp)** models writes as **named mutators** (mutations-as-actions). Local mutations apply instantly; on pull, Replicache rewinds to the last server state, applies the server patch, then **replays pending mutations on top — exactly a git rebase**; the server's execution of the mutator is authoritative ("server reconciliation," borrowed from multiplayer games), and a mutator may legitimately compute a *different* effect during rebase (e.g., the room is taken; book another) [fact:https://doc.replicache.dev/concepts/how-it-works; fact:https://rocicorp.dev/blog/ready-player-two]. Numbers: median ~3ms to run a mutation updating five subscriptions; up to ~64MB local storage; push throughput ceilings around 50 pushes/sec per space under the common strategies; pending mutations are persisted so they survive tab close and accumulate offline [fact:https://doc.replicache.dev/concepts/performance; fact:https://doc.replicache.dev/strategies/global-version]. Note the design: **only unconfirmed mutations are kept as a log; confirmed history is discarded** in favor of server state — a deliberately *finite* event log [fact:https://doc.replicache.dev/concepts/how-it-works].

**Linear's sync engine** (reverse-engineered, endorsed by Linear's CTO): UI code mutates an in-memory MobX object graph; changes become **transactions** in a queue persisted to IndexedDB; the local model is only updated when delta packets return from the server (`/sync/delta`, `lastSyncId` ordering); the queue guarantees no lost writes across crashes [fact:https://github.com/wzhudev/reverse-linear-sync-engine; fact:https://marknotfound.com/posts/reverse-engineering-linears-sync-magic/]. Again: a bounded outbound intent log + server-authoritative state, not an infinite client event log.

**Figma multiplayer** rejected both OT and full CRDTs; it syncs **property-level state deltas** with server-defined ordering — effectively last-writer-wins registers per (object, property), no vector clocks, no tombstone GC; ordering of children uses fractional indexing [fact:https://www.figma.com/blog/how-figmas-multiplayer-technology-works/]. Figma is the strongest counter-example to "log every intent": they ship *state deltas*, not intent events, because the server can order them and LWW-per-property is enough for design documents [fact:https://www.figma.com/blog/how-figmas-multiplayer-technology-works/].

**Automerge/CRDTs** show the storage story at the extreme of "keep all history." Naive op logs are brutal: Automerge 0.14 needed ~1.1GB to represent a 100KB document's history; Kleppmann's columnar encoding compressed op metadata to ~1.1 bytes/op (vs ~55 bytes as JSON), and Automerge 2 stores a 260k-op document *with full history* in ~129KB — ~30% overhead over the raw text [fact:https://automerge.org/blog/automerge-2/; fact:https://github.com/automerge/automerge-perf/blob/master/columnar/README.md; fact:https://josephg.com/blog/crdts-are-the-future/]. Lesson: full-history client logs are viable **only with purpose-built encodings**; a JSON action log grows ~50x faster [estimate, from the 55 vs 1.1 bytes/op figures].

**Synthesis:** every serious sync engine converges on *mutations/events as named, replayable actions* — but they differ on log lifetime: LiveStore keeps the full log as source of truth (and is building compaction because that hurts); Replicache/Linear keep only the *pending* window and let the server own history; Figma keeps no intent log at all. ## 3. Deterministic replay and time travel as a testing superpower

- **Redux was built for this.** Dan Abramov wrote Redux for the React Europe talk "Hot Reloading with Time Travel"; the explicit goal was a minimal-API, fully predictable store enabling "logging, hot reloading, time travel, universal apps, record and replay" [fact:https://blog.isquaredsoftware.com/2017/05/idiomatic-redux-tao-of-redux-part-1/]. Redux DevTools records every dispatched action plus resulting state and lets you step backward/forward by replaying the action list [fact:https://github.com/reduxjs/redux-devtools].
- **Elm makes the same guarantee at the language level:** because effects are data (Cmd/Sub) and update functions are pure, Elm "can have a general purpose time-travel debugger" and test update logic with no setup/teardown [fact:https://guide.elm-lang.org/effects/].
- **rr** proves whole-program deterministic replay is practical: it records nondeterministic syscall results and scheduling, then replays instruction-for-instruction, at ~25% overhead on Firefox-class workloads [fact:https://robert.ocallahan.org/2014/03/introducing-rr.html; fact:https://arxiv.org/pdf/1705.05937]. The transferable insight: **determinism is achieved by capturing every nondeterministic input at the boundary** — which for a Redux-style client means timestamps, random IDs, and network responses must enter as action payloads, never be read inside reducers [assumption, direct corollary].
- **Property-based/model-based testing over reducers is documented practice.** fast-check's model-based testing generates command sequences (up to ~10k commands at `xlarge`) against a system + simplified model, i.e., property-testing a state machine — precisely the reducer shape [fact:https://fast-check.dev/docs/advanced/model-based-testing/]; the Haskell lineage is quickcheck-state-machine [fact:https://hackage.haskell.org/package/quickcheck-state-machine]; a worked web-app example is Concerning Quality's model-based testing writeup [fact:https://concerningquality.com/model-based-testing/].
- **"Record a session as an action log, replay in CI"**: exists but is niche, not standard practice. redux-vcr records/replays user sessions as timestamped Redux actions [fact:https://github.com/joshwcomeau/redux-vcr]; Fowler lists replaying production events into test environments as a core ES debugging benefit [fact:https://martinfowler.com/eaaDev/EventSourcing.html]; commercial "session replay" (LogRocket etc.) captures Redux action+state diffs for triage rather than CI assertion [fact:https://docs.logrocket.com/docs/session-replay]. Verdict: the primitive is proven; a maintained, mainstream "replay action logs in CI" tool is *not* established [estimate].

## 4. The boundary: hexagonal / functional-core–imperative-shell / managed effects

- **Cockburn's ports-and-adapters** (2005): the application core exposes *ports* ("a purposeful conversation"); multiple adapters per port plug in technologies (UI, DB, test harness); the point is to keep business logic uncontaminated by UI/infrastructure and to make the app drivable by tests as just another adapter [fact:https://alistair.cockburn.us/hexagonal-architecture; fact:https://en.wikipedia.org/wiki/Hexagonal_architecture_(software)].
- **Gary Bernhardt's "Boundaries"** (SCNA 2012) argues for *values* as the boundary between subsystems, and "Functional Core, Imperative Shell" demonstrates it: a functional core making all decisions, wrapped by a thin imperative shell doing stdin/stdout/DB/network based on values the core returns [fact:https://www.destroyallsoftware.com/talks/boundaries; fact:https://www.destroyallsoftware.com/screencasts/catalog/functional-core-imperative-shell]. This is the testing rationale for the pattern: isolated unit tests without mocks in the core; few integration tests in the shell.
- **Elm's Cmd/Sub is the purest managed-effects form:** `update` returns *descriptions* of effects (Cmd) and declares interest in inputs (Sub); the runtime executes them, preserving "same input, same output" for all user code [fact:https://guide.elm-lang.org/effects/]. Cmd ≈ driven port invocation as data; Sub ≈ driving port subscription as data [assumption, mapping].
- **Redux middleware/effects map onto ports:** reducers are the functional core; middleware (thunks/sagas/epics) is the imperative shell/adapters where all I/O lives; the Redux style guide's "reducers must not have side effects" is the enforcement of that line [fact:https://redux.js.org/style-guide/]. For a KMP stack: the redux-kotlin store + reducers are platform-free; platform adapters (Ktor, SQLDelight, notification scheduler, clock) sit behind interfaces injected into middleware — Cockburn's test-adapter point means the whole app is exercisable headlessly in CI [assumption, application].

## 5. The "finite set of data-loading abstractions" thesis

Evidence that client data-loading collapses to a small closed set:

- **GraphQL's spec hard-codes exactly three operation types** — query (read), mutation (write), subscription (long-lived push) [fact:https://spec.graphql.org/October2021/#sec-Language.Operations]. An entire industry API surface fits three verbs.
- **TanStack Query's whole model is two primitives + policies:** queries (declarative cached reads, refetched on mount/focus, deduped, GC'd) and mutations (imperative writes with optimistic update/rollback/invalidation); its caching default is stale-while-revalidate — show cached data instantly, revalidate in background [fact:https://tanstack.com/query/latest; fact:https://tanstack.com/query/latest/docs/framework/react/guides/queries]. SWR-the-library is literally named after **HTTP RFC 5861** `stale-while-revalidate`, standardized in 2010 [fact:https://datatracker.ietf.org/doc/html/rfc5861].
- **The read-state shape is also closed:** Kris Jenkins' RemoteData models every remote read as exactly `NotAsked | Loading | Failure e | Success a` — a sum type that eliminates the boolean-flag antipattern, and the pattern has been ported to Angular/TS/Flow/etc. [fact:https://blog.jenkster.com/2016/06/how-elm-slays-a-ui-antipattern/; fact:https://github.com/krisajenkins/remotedata/blob/master/src/RemoteData.elm].
- **Offline-first/sync engines add the remaining two:** pagination (cursor windows over a query) and background sync/outbox (Replicache's persisted pending-mutation queue; Linear's transaction queue) [fact:https://doc.replicache.dev/concepts/how-it-works; fact:https://github.com/wzhudev/reverse-linear-sync-engine].

So the defensible claim: **query (SWR-cached read) + mutation (optimistic write w/ rebase or rollback) + subscription (push) + pagination + outbox/background-sync** covers documented practice across GraphQL, TanStack, Replicache, Linear, and Elm — and each maps cleanly to (action, reducer, middleware-port) triples: e.g. `QueryRequested/QuerySucceeded/QueryFailed` + RemoteData state + a fetch port [fact: URLs above for the primitives; the "that's all there is" universality is [estimate] — well-supported convergent evidence, not a theorem].

## 6. Where the extreme breaks down: what must stay OUT of the log

- **High-frequency ephemeral UI state.** The Redux Style Guide is explicit: most form state should not go in Redux, and "dispatching an action on every keystroke of a text input probably isn't worthwhile" — keep fast-changing local values in component state, dispatch once on commit [fact:https://redux.js.org/style-guide/; fact:https://redux.js.org/faq/organizing-state].
- **Sync-engine practitioners institutionalize the same split.** Yjs puts cursor position/presence in a separate **Awareness** protocol: "Awareness information isn't stored in the Yjs document, as it doesn't need to be persisted"; entries expire after ~30s of silence [fact:https://docs.yjs.dev/getting-started/adding-awareness; fact:https://docs.yjs.dev/api/about-awareness]. Automerge-repo mirrors it with **ephemeral messages**: gossiped to peers, "not persisted to the underlying automerge doc in any way," canonical example: cursors [fact:https://automerge.org/docs/reference/repositories/ephemeral/]. LiveStore's tiering (synced vs client-only events) is the same instinct one level up [fact:https://dev.docs.livestore.dev/reference/concepts/]. Figma keeps live cursors/presence out of the document sync path entirely [fact:https://www.figma.com/blog/how-figmas-multiplayer-technology-works/].
- **Intent events vs state deltas.** Young: the event log's distinguishing property is that it captures **intent** [fact:https://www.kurrent.io/blog/transcript-of-greg-youngs-talk-at-code-on-the-beach-2014-cqrs-and-event-sourcing]. Replicache mutators and LiveStore events are intent (replayable business operations, rebase can recompute effects) [fact:https://doc.replicache.dev/concepts/how-it-works]; Figma ops and Linear delta packets are state deltas (LWW values, no recomputation) [fact:https://www.figma.com/blog/how-figmas-multiplayer-technology-works/]. Intent logs enable smarter merges and audit; delta streams are cheaper and compact trivially (last-write-per-key ≡ Kafka compaction) [fact:https://docs.confluent.io/kafka/design/log_compaction.html; synthesis [assumption]]. During a drag, the *intent* is the final position — emitting 300 intermediate `MOVED_TO` events records noise, not intent [assumption, consistent with the ephemeral-channel designs above].
- **Command vs event semantics.** Commands are imperative requests that **can be rejected**; events are immutable past facts [fact:https://blog.leifbattermann.de/2017/04/21/12-things-you-should-know-about-event-sourcing/; fact:https://www.kurrent.io/blog/transcript-of-greg-youngs-talk-at-code-on-the-beach-2014-cqrs-and-event-sourcing]. Redux blurs this (an "action" is dispatched imperatively but reducers can't reject it); Replicache's rebase semantics restore the distinction: the client's mutation is a command, and the server's authoritative execution decides the resulting event/state [fact:https://rocicorp.dev/blog/ready-player-two]. Naming discipline follows: log `TODO_COMPLETED` (fact), validate `completeTodo` (command) before it reaches the log [assumption].

### Practical rulebook distilled (synthesis, [assumption] unless cited)

1. Two tiers of "action": **durable domain events** (persisted, versioned, synced, upcast-able) vs **ephemeral UI actions** (in-memory only, never serialized) — mirror LiveStore synced/client-only and Yjs doc/awareness.
2. Persisted materialized state = snapshot; replay only tails; don't build snapshotting until replay measurably hurts (Young's ~1k-event heuristic).
3. Compaction by key for latest-value-only facts (Kafka/LiveStore #136).
4. Determinism at the port boundary: clock/random/network enter as action payloads (rr's lesson), enabling DevTools-style time travel, fast-check model-based testing, and redux-vcr-style session replay.
5. Reducers = functional core; middleware = imperative shell over Cockburn ports; effects-as-data (Elm Cmd/Sub) is the ideal to approximate.
6. Data loading needs only ~5 primitives (query/mutation/subscription/pagination/outbox) with RemoteData as the universal read-state type.
